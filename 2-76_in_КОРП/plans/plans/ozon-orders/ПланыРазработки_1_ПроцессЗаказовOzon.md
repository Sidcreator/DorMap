# Планы разработки 1: процесс работы заказов Ozon

> Цель документа: сделать планы внедрения процесса работы заказов Ozon доступными из активной задачи 1 дорожной карты и привязать их к XML-архитектуре проекта `D:\1c\EDT КОРП УОТ\76_in_КОРП`.

---

## Контекст

Активная задача `1` создана на основе результатов анализа `6.8`. Старые запросы `план разработки 6.8` и `план разработки 6.8.PN` являются историческими алиасами к задаче `1`.

`УОТКОРП` рассматривается как следующая версия `УОТ607`: базовые Ozon-артефакты из `УОТ607` нужно переносить не механически, а с адаптацией под расширенную архитектуру КОРП.

### Архитектурные ориентиры

| Область | УОТ607 | УОТКОРП | Решение для внедрения |
|---|---|---|---|
| Интеграция Ozon | `DataProcessors/CG_Online_Ozon_ИнтеграцияСОзон` | `DataProcessors/CG_Online_Ozon_ИнтеграцияСОзон` + `CommonModules/CG_Online_Ozon_ИнтеграцияСОзон` | Формы/макеты адаптировать к общему модулю КОРП |
| Настройка | Есть `Forms/ФормаНастройки` | Есть только `Forms/Форма` | Внедрить `ФормаНастройки`, назначить `DefaultForm` |
| FBO | Есть `Templates/ПоставкаФБО`, `ТипыСкладовСервиса`, `ФункцииСервиса` | Отсутствуют | Перенести XML-макеты и проверить зависимости |
| Статусы поставок | Есть `Enums/CG_Online_Ozon_СтатусыПоставок.xml` | Отсутствует | Создать/перенести перечисление |
| КОРП-расширения | Ограниченный набор | Есть `НастройкиДанныхФайлов`, `СтатусыМодерацииТоваровСервисов`, `ТехнологическиеОперации` | Использовать как точки адаптации новой версии |

## 1.P1. Настройка коннектора Ozon

**Цель:** внедрить мастер настройки Ozon в `УОТКОРП`, чтобы пользователь мог настроить API-ключи, склады, цены и сопоставления без обращения к внутренним XML/регистрам.

| Блок | План разработки |
|---|---|
| XML-артефакты | Взять за основу `УОТ607/DataProcessors/CG_Online_Ozon_ИнтеграцияСОзон/Forms/ФормаНастройки.*` и каталог `Forms/ФормаНастройки/Ext/`. |
| Адаптация | Проверить обращения формы к модулю объекта/менеджера и заменить прямые зависимости на вызовы `УОТКОРП/CommonModules/CG_Online_Ozon_ИнтеграцияСОзон`. |
| Метаданные | В основном XML обработки `CG_Online_Ozon_ИнтеграцияСОзон.xml` назначить `DefaultForm = ФормаНастройки`, если тесты формы успешны. |
| КОРП-учёт | Согласовать настройки с макетом `НастройкиДанныхФайлов` и существующими регистрами/справочниками КОРП. |
| Проверка | Открытие формы, сохранение настроек, повторное открытие, проверка заполнения API-параметров и складов. |

**Критерий готовности:** `ФормаНастройки` открывается в КОРП, сохраняет параметры и не вызывает отсутствующие методы `УОТ607`.

## 1.P2. Авторизация

**Цель:** обеспечить стабильную авторизацию Ozon через настройки КОРП и единый модуль интеграции.

| Блок | План разработки |
|---|---|
| XML-артефакты | Проверить существующие объекты авторизации в `УОТКОРП`: обработка Ozon, общий модуль Ozon, общие настройки сервисов. |
| Адаптация | Разделить хранение токенов/ключей и выполнение HTTP-запросов: форма только сохраняет, общий модуль выполняет авторизацию. |
| Совместимость | Сверить с `УОТ607` вызовы авторизации в `ObjectModule.bsl`, `ManagerModule.bsl`, `Forms/ФормаНастройки/Ext/Form/Module.bsl`. |
| Безопасность | Не переносить устаревшие прямые обращения к API, если в КОРП уже есть общий HTTP/логирующий слой. |
| Проверка | Тестовый запрос к Ozon, обработка ошибки авторизации, запись диагностического сообщения. |

**Критерий готовности:** авторизация выполняется через КОРП-слой интеграции, а ошибки видны пользователю и диагностике.

## 1.P3. Загрузка товаров

**Цель:** использовать существующую КОРП-архитектуру для загрузки товаров, свойств и категорий Ozon без отката к старой логике `УОТ607`.

| Блок | План разработки |
|---|---|
| XML-артефакты | Проверить макеты `ВерсииМетодовAPI`, `КомандыСервиса`, `ТипыЦенСервисов` в обеих поставках. |
| Адаптация | Сохранить КОРП-макеты как приоритетные, переносить из `УОТ607` только отсутствующие данные команд/API. |
| Данные | Привязать загрузку к справочникам и регистрам КОРП, не создавать дублей объектов. |
| Версионность API | Сверить версии методов Ozon и отметить расхождения в макете `ВерсииМетодовAPI`. |
| Проверка | Загрузка категорий, карточек, свойств, повторная загрузка без дублей. |

**Критерий готовности:** товары и свойства загружаются через существующую КОРП-модель данных.

## 1.P4. Выгрузка каталога

**Цель:** подготовить выгрузку остатков, цен, фото и габаритов в Ozon с учётом расширенной КОРП-структуры.

| Блок | План разработки |
|---|---|
| XML-артефакты | Использовать существующие в `УОТКОРП` макеты и модули Ozon/XML/YML; сверить с `УОТ607` только недостающие команды. |
| Остатки | Согласовать источники остатков с регистрами КОРП и моделями работы сервисов. |
| Цены | Проверить `ТипыЦенСервисов` и режимы выгрузки цен. |
| Медиа | Проверить обработку фото и габаритов через существующие КОРП-модули файлов/номенклатуры. |
| Проверка | Тестовая выгрузка одного товара, группы товаров, повторная выгрузка с изменениями. |

**Критерий готовности:** каталог выгружается без нарушения текущей КОРП-логики обменов.

## 1.P5. Загрузка заказов

**Цель:** загрузить заказы Ozon в КОРП-регистры и документы с сохранением расширенной аналитики КОРП.

| Блок | План разработки |
|---|---|
| XML-артефакты | Опора на существующие `CG_Online_Ozon_СтатусыЗаказов`, регистры заказов клиентов в сервисах и модуль Ozon КОРП. |
| Маппинг | Сопоставить статусы Ozon с внутренними статусами КОРП и режимами формирования заказов клиентов. |
| Документы | Проверить создание/обновление заказа клиента, связь с сервисом, складом, моделью работы. |
| Идемпотентность | Повторная загрузка не должна создавать дубли заказов. |
| Проверка | Загрузка новых, изменённых и отменённых заказов. |

**Критерий готовности:** заказы Ozon попадают в КОРП-цепочку документов и регистров без дублей.

## 1.P6. Обработка заказов

**Цель:** обеспечить сборку, лист подбора, маркировку и подготовку заказов Ozon к отгрузке.

| Блок | План разработки |
|---|---|

| Адаптация | Использовать существующие КОРП-обработки `CG_Online_АРМКладовщика`, `CG_Online_ПечатьЗаказовКлиентовСервисов`, `CG_Online_ГрупповаяМаркировка`. |

| Проверка | Сборка заказа, печать листа, маркировка, переход к отгрузке. |

**Критерий готовности:** заказ проходит внутреннюю обработку до готовности к отгрузке.

## 1.P7. Отгрузка FBO

**Цель:** закрыть главный разрыв FBO в `УОТКОРП`: внедрить поставки, типы складов, функции сервиса и статусы поставок.

| Блок | План разработки |
|---|---|
| XML-артефакты | Перенести/создать `Templates/ПоставкаФБО`, `Templates/ТипыСкладовСервиса`, `Templates/ФункцииСервиса`, `Enums/CG_Online_Ozon_СтатусыПоставок.xml`. |
| Адаптация | Проверить UUID, имена, ссылки на обработку Ozon, сериализацию табличных макетов и использование в модулях. |
| Интеграция | Подключить FBO-функции к `CommonModules/CG_Online_Ozon_ИнтеграцияСОзон`, не дублируя бизнес-логику в форме. |
| КОРП-данные | Учитывать `CG_Online_АРМУправлениеПоставкамиСоСкладаСервиса`, регистры поставок и модели работы сервисов. |
| Проверка | Создание поставки FBO, выбор склада Ozon, отправка/обновление статуса поставки. |

**Критерий готовности:** FBO-поставка создаётся и проходит статусную цепочку в КОРП.

**Цель:** синхронизировать статусы заказов и поставок Ozon с внутренними статусами КОРП.

| Блок | План разработки |
|---|---|

## 1.P10. Финансы

**Цель:** довести цепочку Ozon до финансового закрытия: отчёты комиссионера, комиссии, скидки, сверка.

| Блок | План разработки |
|---|---|
| XML-артефакты | Проверить отчёты, документы и регистры КОРП по комиссионерам/финансовой аналитике сервисов. |
| Данные Ozon | Согласовать загрузку финансовых операций с заказами, возвратами и поставками. |
| Аналитика | Использовать расширенную аналитику КОРП, не переносить упрощённую модель `УОТ607`, если она конфликтует. |
| Сверка | Предусмотреть сверку сумм заказа, комиссии, скидок, возвратов и выплат. |
| Проверка | Формирование финансового отчёта по периоду, сверка по одному заказу и по группе заказов. |

**Критерий готовности:** заказ Ozon проходит путь от загрузки до финансового закрытия.

## Общие критерии приёмки

- Все XML-объекты проходят загрузку в EDT без ошибок структуры.
- Не появляются дубли объектов с одинаковым назначением.
- Новые/перенесённые объекты используют префикс `CG_Online_Ozon_`.
- Логика формы настройки не дублирует общий модуль `CG_Online_Ozon_ИнтеграцияСОзон`.
- Цепочка Ozon работает как следующая версия `УОТ607`, но с КОРП-регистрами, справочниками и общими модулями.
