Причина: лишний CROSS JOIN в НовыеЗаказы

1. Симптом

  • Загрузка заказов Ozon через тест проходит без ошибок
  • ДозаполнитьЗаказыКлиентовСервисов() возвращает пустой ЗаказыКСозданию
  • Документы CG_Online_ЗаказКлиентаСервиса — не создаются
  • Регистры CG_Online_ЗаказыКлиентовСервисов и CG_Online_РеестрДокументовСервисов — пусты

3.2. Ключевое отличие УОТ609 vs УОТКОРП в НовыеЗаказы

⚠️ Найден дефект: лишний CROSS JOIN со СтатусОшибка

В УОТКОРП во временную таблицу НовыеЗаказы добавлен CROSS JOIN с СтатусОшибка, которого не было в УОТ609. Если справочник CG_Online_СтатусыДокументовСервисов не содержит элемента с ТипСтатуса = Ошибка — CROSS JOIN даёт 0 строк, и НовыеЗаказы (а значит и ЗаказыКСозданию) будут пусты.

7. Вывод

🔴 Первопричина

Дефект рефакторинга при переносе логики из УОТ609 в УОТКОРП. Во временную таблицу НовыеЗаказы (строка 3791 модуля CG_Online_ЗаказКлиентаСервиса) добавлен лишний CROSS JOIN с СтатусОшибка. В оригинале (УОТ609) этот CROSS JOIN присутствует только в пакете [7] (ЗаказыСОшибкойТоваров), где он необходим для простановки статуса «Ошибка» несопоставленным товарам. В НовыеЗаказы он не нужен — эта таблица должна содержать все заказы без существующего документа ЗКС, независимо от наличия статуса Ошибка.

7.1. Условия проявления

  • Триггер: отсутствие элемента справочника CG_Online_СтатусыДокументовСервисов с ТипСтатуса = Ошибка на момент выполнения запроса
  • В тестовом контуре: если тест ЗагрузитьНастройкиАвторизации() не выполнился или не создал статус Ошибка
  • На чистой базе: при первом запуске загрузки заказов до загрузки справочника статусов

7.2. Исправление

Убрать CROSS JOIN со СтатусОшибка из запроса создания НовыеЗаказы (строка 3791). Источником данных должна остаться только СопоставленныеЗаказыСервисов, как в оригинальном УОТ609:

ПОМЕСТИТЬ НовыеЗаказы ИЗ СопоставленныеЗаказыСервисов КАК СопоставленныеЗаказыСервисов // без , СтатусОшибка ГДЕ (СопоставленныеЗаказыСервисов.ЗаказКлиентаСервиса = &ПустойЗаказКлиентаСервиса ИЛИ СопоставленныеЗаказыСервисов.ЗаказКлиентаСервиса ЕСТЬ NULL ИЛИ СопоставленныеЗаказыСервисов.ЗаказКлиентаСервиса.ПометкаУдаления)

8.3. Инициализация в тесте

  • Тест ЗагрузитьНастройкиАвторизации() (строка 538, КОРПтест/ВызовСервера/Module.bsl) вызывает ЗагрузитьСтатусыДокументовСервисов()
  • Та вызывает CG_Online_СтатусыДокументовСервисов.ЗагрузитьВсеОбъекты(), который через макеты ИСТП загружает статусы включая Ошибка
  • Порядок тестов имеет значение: если ЗагрузитьЗаказКлиентаСервиса() запускается без предварительного ЗагрузитьНастройкиАвторизации() — СтатусОшибка может отсутствовать

9. Рекомендации

  1. Исправить код: убрать , СтатусОшибка КАК СтатусОшибка из FROM в запросе НовыеЗаказы (строка 3791 CG_Online_ЗаказКлиентаСервиса/Ext/Module.bsl).
  2. Проверить тестовое окружение: убедиться, что тест ЗагрузитьНастройкиАвторизации() гарантированно выполняется перед ЗагрузитьЗаказКлиентаСервиса() (порядок в тестовом наборе).
  3. Добавить проверку: после вызова ДозаполнитьЗаказыКлиентовСервисов() логировать размер ЗаказыКСозданию для диагностики.
  4. Регрессионный тест: добавить тест, проверяющий, что ЗаказыКСозданию заполняется даже при отсутствии статуса Ошибка в справочнике.
